# Swagger / OpenAPI

If you created your project with `drp init myapp --auth`, you already have
Swaggo (Swagger for Go) wired up — annotations on your handlers, the
`/swagger/` endpoint registered, and a `docs/docs.go` stub in place.

---

## Step 1 — Run your API

```bash
drp run api
```

Once the server is listening, open your browser at:

```
http://localhost:8080/swagger/
```

You should see the Swagger UI. It will show the metadata (title, version,
host) but **no endpoints yet** if you haven't regenerated the docs after
generating the project.

---

## Step 2 — Install the swag CLI

To generate the real endpoint documentation you need the `swag` binary:

```bash
go install github.com/swaggo/swag/cmd/swag@latest
```

---

## Step 3 — Regenerate the docs

From your project root run:

```bash
swag init
```

This parses all the `// @Summary`, `// @Router`, `// @Param`, … annotations
in your source code and produces three files:

| File | What it contains |
|---|---|
| `docs/docs.go` | Full `swag.Spec` with all paths, definitions, and security schemes |
| `docs/swagger.json` | OpenAPI 2.0 spec as JSON |
| `docs/swagger.yaml` | Same spec as YAML |

Now restart your API and refresh the Swagger UI — you will see all 5
endpoints listed:

| Method | Path | Description |
|---|---|---|
| POST | `/auth/register` | Register a new user |
| POST | `/auth/login` | Authenticate and receive tokens |
| POST | `/auth/refresh` | Refresh an expired access token |
| POST | `/auth/logout` | Revoke tokens |
| GET | `/me` | Get the authenticated user's profile |

Every route includes its request body schema, response schema, and expected
status codes.

---

## How the annotations work

**Package-level** — `internal/auth/docs.go` sets the API metadata:

```go
// @Title drp API
// @Description API generated by drp-go-cli
// @Version 1.0
// @Host localhost:8080
// @BasePath /
```

**Handler-level** — each endpoint has `@Summary`, `@Tags`, `@Accept`,
`@Produce`, `@Param`, `@Success`, `@Failure` and `@Router`. Example from
`internal/auth/handler.go`:

```go
// Register handles POST /auth/register
//
// @Summary Register a new user
// @Tags auth
// @Accept json
// @Produce json
// @Param request body user.RegisterRequest true "Registration payload"
// @Success 201 {object} map[string]any
// @Failure 400 {object} map[string]string
// @Router /auth/register [post]
```

**Security** — the `/me` endpoint uses `@Security BearerAuth` so the
Swagger UI includes an "Authorize" button where you can paste your access
token.

---

## Customising the docs

Edit `internal/auth/docs.go` to change:

| Annotation | What it controls |
|---|---|
| `@Title` | The API title shown in the Swagger UI header |
| `@Description` | A short description below the title |
| `@Version` | API version string |
| `@Host` | The host and port the API runs on |
| `@BasePath` | URL prefix for all routes (default `/`) |

After editing, run `swag init` again to regenerate.

---

## Re-generating after adding new endpoints

Whenever you add a new handler with Swaggo annotations (or modify an existing
one), run:

```bash
swag init
```

This picks up your changes and updates the three `docs/` files. There is no
need to restart the API — the swagger UI reads from the generated Go code
at compile time, so you must rebuild.

---

## The `docs/docs.go` stub

When you first ran `drp init`, the generated `docs/docs.go` is a minimal stub:

```go
var SwaggerInfo = &swag.Spec{
    Version:   "1.0",
    Host:      "localhost:8080",
    BasePath:  "/",
    Title:     "drp API",
    Description: "API generated by drp-go-cli",
}
func init() {
    swag.Register(SwaggerInfo.InstanceName(), SwaggerInfo)
}
```

This exists so `cmd/api/main.go` compiles (it imports `_ "yourmodule/docs"`).
Without running `swag init` the Swagger UI will show metadata but no routes.
Run `swag init` once to replace the stub with the full spec.
